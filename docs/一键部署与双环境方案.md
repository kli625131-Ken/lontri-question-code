# 问题库运维平台：一键部署与双环境方案

本文档用于把项目部署到公司服务器上的 Ubuntu 虚拟机，并在同一台 Ubuntu 中运行两套环境：

- 生产环境：正式使用
- 测试环境：上线前验证

当前推荐方案不是复制两台虚拟机，而是：

```text
一台 Ubuntu 虚拟机
├── /opt/questioncode-prod    生产环境
└── /opt/questioncode-test    测试环境
```

## 1. 为什么不建议复制虚拟机

复制虚拟机可以做，但不是当前最优方案。

不推荐复制虚机的原因：

- 两台虚机要占两份内存和磁盘。当前 Ubuntu 已分配 8GB，复制后资源压力更大。
- 两台虚机都要维护 Ubuntu、Docker、镜像源、网络、SSH、备份。
- 克隆后 IP、主机名、SSH 指纹可能冲突，需要额外处理。
- 项目本身已经用 Docker 隔离，生产和测试放在同一台 Ubuntu 内完全够用。

推荐方案：

```text
生产环境：http://192.168.0.90
测试环境：http://192.168.0.90:8081
```

只有在以下情况才建议复制虚拟机：

- 公司要求生产和测试必须操作系统级隔离。
- 测试环境要安装大量不确定软件，可能污染系统。
- 两套环境需要完全不同的 Ubuntu/Docker/MySQL 版本。
- 服务器资源充足，并且后续有人专门维护两台虚机。

## 2. 两套 Docker 的隔离方式

每套环境有独立目录、独立配置、独立端口、独立数据库卷、独立上传文件目录。

| 项目 | 生产环境 | 测试环境 |
|---|---|---|
| 目录 | `/opt/questioncode-prod` | `/opt/questioncode-test` |
| 前端访问 | `http://192.168.0.90` | `http://192.168.0.90:8081` |
| 后端端口 | `127.0.0.1:8082` | `127.0.0.1:18082` |
| MySQL 端口 | `127.0.0.1:3307` | `127.0.0.1:13307` |
| Compose 项目名 | `questioncode_prod` | `questioncode_test` |
| 容器名前缀 | `questioncode-prod` | `questioncode-test` |
| 上传文件 | `/opt/questioncode-prod/uploads` | `/opt/questioncode-test/uploads` |

容器示例：

```text
questioncode-prod-mysql
questioncode-prod-backend
questioncode-prod-frontend

questioncode-test-mysql
questioncode-test-backend
questioncode-test-frontend
```

## 3. 已加入的自动化脚本

本项目新增两个脚本：

```text
scripts/package-deploy.ps1
scripts/server-oneclick-deploy.sh
```

用途：

| 脚本 | 在哪里执行 | 用途 |
|---|---|---|
| `scripts/package-deploy.ps1` | 本机 Windows | 生成部署 zip |
| `scripts/server-oneclick-deploy.sh` | Ubuntu 服务器 | 自动部署 prod/test |

## 4. 本机打包

在本机项目目录执行：

```powershell
cd D:\code\lontri\questioncode
powershell -ExecutionPolicy Bypass -File .\scripts\package-deploy.ps1
```

脚本会生成：

```text
backup\questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

如果首次部署需要带上传文件：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\package-deploy.ps1 -IncludeUploads
```

如果要指定数据库 SQL：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\package-deploy.ps1 -DbSqlPath .\backup\problem_db_deploy.sql -IncludeUploads
```

说明：

- 默认不包含 `node_modules`、`target`、`dist`、日志文件等临时内容。
- 默认会复制代码、Docker 配置、文档、脚本、SQL 目录。
- 如果存在 `backup/problem_db_deploy.sql`，会自动带入部署包。

## 5. 上传部署包到服务器

在公司电脑 PowerShell 中执行，文件名按实际生成结果修改：

```powershell
scp "D:\code\lontri\questioncode\backup\questioncode-deploy-YYYYMMDD_HHMMSS.zip" lontri@192.168.0.90:/home/lontri/
```

如果部署脚本还没有在服务器上，可以先从压缩包中解压出脚本，或者直接把脚本也上传：

```powershell
scp "D:\code\lontri\questioncode\scripts\server-oneclick-deploy.sh" lontri@192.168.0.90:/home/lontri/
```

## 6. 首次部署测试环境

建议先部署测试环境。

登录服务器：

```powershell
ssh lontri@192.168.0.90
```

执行：

```bash
bash /home/lontri/server-oneclick-deploy.sh test /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

```text

```

```bash
cd /opt/questioncode-test

```

部署完成后访问：

```text

检查：

```

## 8. 从当前单环境迁移到双环境

如果服务器上现在是：

```text
/opt/questioncode
```

建议按以下方式迁移：

```bash
cd /opt/questioncode
mkdir -p backup
docker compose exec -T mysql sh -c 'mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" --default-character-set=utf8mb4 problem_db' > backup/problem_db_before_split_$(date +%F_%H%M%S).sql
tar -czf backup/uploads_before_split_$(date +%F_%H%M%S).tar.gz uploads

docker compose down

cd /opt
mv questioncode questioncode-prod
```

然后部署测试环境：

```bash
bash /home/lontri/server-oneclick-deploy.sh test /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

生产环境如果需要更新为新脚本支持的配置，再执行：

```bash
bash /home/lontri/server-oneclick-deploy.sh prod /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

## 9. 以后更新代码的流程

标准流程：

```text
本机改代码
→ 本机打包
→ 上传服务器
→ 先部署测试环境
→ 浏览器验证测试环境
→ 再部署生产环境
→ 验证生产环境
```

测试环境更新：

```bash
bash /home/lontri/server-oneclick-deploy.sh test /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

生产环境更新：

```bash
bash /home/lontri/server-oneclick-deploy.sh prod /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

脚本会自动做这些事：

- 如果目标环境已经存在，会先备份数据库和 uploads。
- 停止当前环境。
- 把旧目录挪到 `/opt/questioncode-prod-old-时间` 或 `/opt/questioncode-test-old-时间`。
- 解压新代码。
- 保留旧 `.env`。
- 默认保留旧 `uploads`。
- 重新构建并启动 Docker。
- 等待 MySQL 和后端健康检查。

## 10. 数据库导入策略

脚本默认：

```text
IMPORT_DB=auto
```

含义：

- 首次部署时，如果部署包内有 `backup/problem_db_deploy.sql`，自动导入。
- 已存在环境更新时，不自动覆盖数据库。

如果明确要强制导入数据库：

```bash
IMPORT_DB=always bash /home/lontri/server-oneclick-deploy.sh test /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

如果明确不要导入数据库：

```bash
IMPORT_DB=never bash /home/lontri/server-oneclick-deploy.sh test /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

生产环境不要轻易使用 `IMPORT_DB=always`，这会覆盖当前生产数据库。

## 11. 上传文件策略

脚本默认：

```text
PRESERVE_UPLOADS=true
```

含义：

- 环境已经存在时，保留原来的 `uploads`。
- 新包里的 uploads 不会覆盖已有上传文件。

如果测试环境想完全使用部署包里的 uploads：

```bash
PRESERVE_UPLOADS=false bash /home/lontri/server-oneclick-deploy.sh test /home/lontri/questioncode-deploy-YYYYMMDD_HHMMSS.zip
```

生产环境建议保持默认，不要覆盖 uploads。

## 13. 回滚

每次部署后，旧目录会被移动为：

```text
/opt/questioncode-prod-old-YYYY-MM-DD_HHMMSS
/opt/questioncode-test-old-YYYY-MM-DD_HHMMSS
```

如果新版本有问题，以生产环境为例：

```bash
cd /opt/questioncode-prod
docker compose down

cd /opt
mv questioncode-prod questioncode-prod-bad-$(date +%F_%H%M%S)
mv questioncode-prod-old-对应时间 questioncode-prod

cd /opt/questioncode-prod
docker compose up -d
docker compose ps
curl http://127.0.0.1:8082/actuator/health
```

如果数据库也被错误修改，需要用部署前自动生成的 SQL 备份恢复。

## 14. 禁止随便执行的命令

不要在生产环境随便执行：

```bash
docker compose down -v
```

原因：

- `-v` 会删除 Docker 数据卷。
- MySQL 数据在数据卷中。
- 这可能导致生产数据库被清空。

可以执行：

```bash
docker compose down
```

这只停止和删除容器，不删除数据库卷。

## 15. 验收清单

每次部署后检查：

```bash
docker compose ps
curl http://127.0.0.1:8082/actuator/health      # 生产
curl http://127.0.0.1:18082/actuator/health     # 测试
```

浏览器检查：

- 生产：`http://192.168.0.90`
- 测试：`http://192.168.0.90:8081`
- 能登录
- 问题台账能打开
- 图片附件能打开
- 新增/上传功能至少测一次

## 16. 推荐执行顺序

当前阶段建议按以下顺序落地：

1. 本机执行 `scripts/package-deploy.ps1` 生成新部署包。
2. 上传部署包和 `server-oneclick-deploy.sh` 到服务器。
3. 先部署测试环境。
4. 确认测试环境可访问、图片正常。
5. 再把当前生产整理为 `/opt/questioncode-prod`。
6. 后续所有代码更新先走测试，再走生产。
