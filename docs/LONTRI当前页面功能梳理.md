# LONTRI 当前页面功能梳理

> 适用版本：当前仓库 `frontend/src/views`、`frontend/src/router` 对应页面实现  
> 输出目的：梳理当前平台已经具备的页面能力，明确“现阶段能做什么、做到哪一步、还有哪些边界”

---

## 1. 页面总体结构

当前前端采用 `登录页 -> 主布局 -> 各业务页面` 的结构。

主导航共 4 个核心业务页面：

1. 运营总览
2. 问题台账
3. 导入中心
4. 项目视图

此外还有：

1. 登录页
2. 问题详情页
3. 401 无权限页

## 4. 各页面详细说明

## 4.1 登录页

路由：
- `/login`

页面定位：
- 平台认证入口
- 用户登录后进入业务系统

当前功能：
- 输入用户名和密码
- 支持回车登录
- 登录成功后写入 token
- 登录成功后写入当前用户信息
- 支持根据 `redirect` 参数跳回原本要访问的页面
- 页面内展示开发环境默认账号：`admin / admin123`

对应接口：
- `POST /auth/login`
- `GET /auth/current-user`
- `POST /auth/logout`

现阶段说明：
- 页面有“记住登录状态”勾选项，但从现有实现看，主要依赖 token 存储，未看到更细的长期会话差异化处理
- “忘记密码”目前是展示按钮，未见实际找回流程

---

## 4.2 运营总览

路由：
- `/dashboard`

页面定位：
- 首页工作台
- 将“问题总览、导入批次、项目状态”集中到一个入口页

当前功能：
- 展示问题总量
- 展示待处理问题数
- 展示处理中问题数
- 展示已关闭问题数
- 展示逾期提醒数
- 展示待人工确认导入数
- 展示最近问题列表
- 点击最近问题可进入问题详情页
- 展示最新导入批次摘要
- 展示批次状态、总行数、待确认数、已入库数、已跳过数
- 展示批次按行类型、清洗状态的统计标签
- 展示项目卡片列表
- 展示项目开口问题数、总问题数、提醒阈值
- 从首页跳转到问题台账、导入中心、项目视图

对应接口：
- `GET /issues/overview`
- `GET /issues`
- `GET /imports`
- `GET /projects`

价值判断：
- 这是当前平台的“运营驾驶舱”
- 更偏业务入口页，而不是深分析型仪表盘

现阶段边界：
- 当前偏总览与跳转，不是完整多图表 BI 看板
- 还没有日报、周报、月报、年报独立页面
- 还没有趋势图、同比环比、根因分析等深度分析模块

---

## 4.3 问题台账

路由：
- `/issues`

页面定位：
- 平台核心业务页面
- 用于集中管理问题主单

当前功能：
- 按项目筛选
- 按当前状态筛选
- 按闭环状态筛选
- 按负责人筛选
- 按分类关键词筛选
- 按日期范围筛选
- 按关键字搜索问题描述、处理进展、备注
- 展示分页问题列表
- 查看问题所属项目
- 查看问题描述
- 查看分类路径
- 查看当前状态
- 查看闭环状态
- 查看负责人
- 查看优先级
- 查看发现时间
- 查看提醒状态或提醒阈值
- 查看最新进展
- 点击行跳转问题详情
- 通过“新建问题”弹窗创建问题

新建问题弹窗当前支持：
- 选择项目
- 填写发现时间
- 填写来源
- 填写分类
- 填写负责人
- 填写优先级
- 填写严重程度
- 设置提醒天数
- 填写问题描述
- 填写影响范围
- 填写最新进展
- 填写备注

对应接口：
- `GET /issues`
- `GET /issues/overview`
- `POST /issues`
- `GET /projects`

价值判断：
- 这是当前平台的“问题主台账”
- 已具备问题列表、检索、筛选、录入的基础闭环

现阶段边界：
- 当前筛选能力已经具备基础实用性，但还不是全文检索系统
- 没有批量编辑、批量关闭、批量导出等高级台账动作
- 没看到独立的 SLA 管理、超时升级、责任流转机制

## 4.4 问题详情页

路由：
- `/issues/:id`

页面定位：
- 单问题全链路维护页
- 支撑问题跟进、闭环、重开、经验复用

当前功能：
- 查看问题标题
- 查看所属项目
- 查看来源
- 查看当前状态标签
- 查看闭环状态标签
- 查看分类
- 查看发现时间
- 查看负责人
- 查看优先级和严重程度
- 查看提醒阈值与是否逾期
- 查看影响范围
- 查看最新进展
- 查看备注
- 查看来源 Sheet
- 查看来源行号
- 查看完成时间

处理动作：
- 追加处理记录
- 问题闭环
- 闭环后重新打开

页面还包含：
- 处理时间线
- 每条时间线展示动作类型、处理内容、操作人、状态流转
- 相似历史问题列表
- 点击相似问题跳转对应详情

对应接口：
- `GET /issues/:id`
- `POST /issues/:id/records`
- `POST /issues/:id/close`
- `POST /issues/:id/reopen`

价值判断：
- 这是当前最接近“问题闭环管理”的页面
- 已经具备从问题产生到持续跟进再到闭环/重开的基础能力

现阶段边界：
- 当前已支持“相似历史问题”，但还没有更强的知识库结构化沉淀页
- 还没有独立根因分析、责任分析、处理标准 SOP 模块
- 目前更像单问题工作台，不是完整 RCA 专题页

## 4.5 导入中心

路由：
- `/imports`

页面定位：
- 历史 Excel 运维数据进入平台的核心入口
- 当前平台最有代表性的能力页面之一

当前功能：
- 上传 `.xlsx` 文件
- 生成导入预览批次
- 展示最近导入批次列表
- 展示每个批次的状态、总行数、待确认数、创建时间
- 选择批次查看详情
- 查看批次摘要信息
- 查看行类型统计
- 查看清洗确认统计
- 查看各 Sheet 分布
- 过滤查看全部行 / 待确认行 / 问题主单行
- 查看每行的：
  - 来源 Sheet
  - 行号
  - 行类型
  - 确认状态
  - 提交状态
  - 摘要
  - 清洗备注
- 编辑单行标准化结果
- 编辑行的确认状态
- 编辑行的确认备注
- 查看原始行快照
- 提交整批数据入库

对应接口：
- `POST /imports/excel`
- `GET /imports`
- `GET /imports/:id`
- `PUT /imports/:batchId/rows/:rowId`
- `POST /imports/:id/commit`

价值判断：
- 这是当前系统和你业务场景最贴合的页面
- 它已经不是简单上传页，而是“导入预览 + 清洗确认 + 批次提交”的完整链路

现阶段边界：
- 当前是“人工辅助确认型”导入，不是完全自动化数据治理平台
- 还没有独立的数据质量大盘
- 还没有版本化口径管理、映射字典维护页面、批次回滚等更强治理能力

## 6. 当前页面能力对应的业务链路

从页面实现看，当前平台已经形成了一个比较完整的基础业务链路：

```text
登录
-> 进入运营总览
-> 从导入中心上传 Excel
-> 生成导入批次与待确认数据
-> 人工修正清洗结果
-> 提交批次入库
-> 形成问题台账
-> 在问题详情中持续跟进
-> 闭环或重开问题
-> 从项目视图辅助查看联系人与质保信息
```

这说明当前平台的重点不在“花哨展示”，而在“历史运维数据接入 + 问题台账化 + 闭环追踪”。

## 9. 对当前页面体系的定位判断

如果用产品阶段来定义，当前平台更接近：

`数据接入型运维问题平台 V1`

它的重点已经比较清晰：

1. 把历史 Excel 运维记录变成结构化问题数据
2. 把问题做成可维护、可闭环、可追溯的主台账
3. 把项目联系人、质保等辅助信息挂接进来
4. 让后续的报表、检索、定位能力有数据基础

所以从页面层面看，当前 V1 最成熟的不是大屏，而是：

- 导入中心
- 问题台账
- 问题详情

这三块是当前系统真正的业务主线。

## 11. 结论

当前页面体系已经具备一个比较完整的基础运维问题平台雏形，重点不是“展示很多图”，而是完成了下面这条主线：

`Excel 导入 -> 清洗确认 -> 问题台账 -> 问题详情 -> 处理记录 -> 闭环追踪 -> 项目辅助信息查看`

从产品成熟度来看，当前最强的能力是“历史运维数据台账化”和“问题闭环追踪”，而不是报表分析和链路诊断。

如果后续继续规划，建议在现有页面基础上优先补齐：

1. 数据治理页面
2. 报表中心页面
3. 问题链路定位页面
4. 知识库页面

这样会和你前面提出的整体建设路线最一致。
